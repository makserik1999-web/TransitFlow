# TransitFlow Mangystau — спецификация MVP для хакатона

> Этот файл кладётся в корень репозитория. Claude Code использует его как главный источник правды о проекте.
> Дедлайн сдачи: 12 июня, 09:00 (презентация + GitHub + ссылка на задеплоенный MVP).

## 1. Что мы строим (одним абзацем)

Веб-платформа координации автотранзита Мангистауской области: грузоотправитель создаёт заявку и видит загруженность узлов в реальном времени, перевозчик берёт заявку и бронирует тайм-слот прохождения узла (порт / погранпереход) с QR-пропуском, аналитик акимата видит сводный дашборд транзита с картой, графиками и AI-сводкой. Одна база данных — три роли — один деплой.

Позиционирование (ответ на вопрос жюри «государство уже цифровизирует логистику»):
существующие системы (АСТАНА-1, цифровизация КТЖ, портовые ИС) закрывают таможню,
ж/д и контейнерные поезда. Мы закрываем нижний слой, который никто не трогает:
3 000+ региональных автоперевозчиков и малый бизнес, которые сегодня ищут друг
друга через звонки и WhatsApp и стоят в живых очередях. Мы — координационный слой
«последней мили» для авто-транзита, а не замена госсистем; архитектура готова к
интеграции с ними через API.

## 2. Роли и ключевые сценарии (это и есть сценарий демо на питче)

1. **Грузоотправитель (shipper)**
   - Регистрируется / логинится (демо-аккаунт уже в сиде).
   - Видит дашборд загруженности узлов: «Порт Актау — высокая загрузка до 14:00, окно после 18:00».
   - Создаёт заявку: тип груза, тоннаж, откуда → куда, желаемая дата.
   - Видит статус заявки: открыта → принята перевозчиком → слот забронирован → выполнена.

2. **Перевозчик (carrier)**
   - Видит ленту открытых заявок, фильтр по направлению/дате.
   - Принимает заявку.
   - Бронирует тайм-слот на узле (например, погранпереход «Темир-Баба», 11 июня 16:00–17:00).
   - Получает QR-пропуск брони (публичная страница: машина, груз, слот, статус). QR показывается на питче — судья может отсканировать со сцены.

3. **Аналитик акимата (analyst)**
   - Дашборд: карта области (Leaflet) с узлами и их текущей загрузкой (цвет = загрузка).
   - Графики: объём транзита по дням (90 дней истории), разбивка по типам груза, по направлениям, по узлам.
   - Прогноз загруженности на 7 дней вперёд (по историческим паттернам).
   - Кнопка «AI-сводка»: Claude API получает агрегаты и возвращает текстовую сводку аномалий и рекомендаций на русском.

## 4. Архитектура

```
монорепо
├── SPEC_TransitFlow.md        ← этот файл
├── backend/                   ← FastAPI + SQLAlchemy + PostgreSQL
│   ├── app/
│   │   ├── main.py            ← FastAPI app, раздаёт и API, и статику фронта
│   │   ├── models.py          ← SQLAlchemy модели
│   │   ├── schemas.py         ← Pydantic схемы
│   │   ├── auth.py            ← JWT, роли
│   │   ├── routers/           ← requests, slots, bookings, analytics, auth
│   │   ├── forecast.py        ← прогноз: среднее по (день недели, час) + сглаживание
│   │   └── ai_summary.py      ← вызов Claude API (anthropic sdk), graceful fallback
│   ├── seed.py                ← генератор синтетики (90 дней истории + демо-юзеры)
│   └── requirements.txt
└── frontend/                  ← React + Vite + Leaflet + Recharts, build → backend/static
```

Решения и почему:
- **Один деплой**: FastAPI раздаёт собранный фронт как статику → один URL, ноль CORS-боли, проще дедлайн. Деплой на Railway (Postgres там же) или Render.
- **Auth**: JWT, поле role у пользователя. Без регистрации по email-подтверждению — демо-аккаунты в сиде (shipper@demo / carrier@demo / analyst@demo, пароль demo123). Кнопки «войти как…» на странице логина для скорости демо.
- **Прогноз** — честно простой: для каждого узла среднее количество машин по (день недели × час) за 90 дней синтетики + множитель тренда. На питче называем это «бейзлайн-модель, архитектура готова к замене на ML при реальных данных». Не выдавать за ML.
- **AI-сводка**: один эндпоинт POST /api/analytics/ai-summary → собирает агрегаты (топ-узлы, отклонения от прогноза, рост/падение по типам груза) → промпт в Claude API → текст. Обязателен fallback: если ключа/кредитов нет, эндпоинт возвращает заранее сгенерированную сводку из БД (флаг is_cached) — демо не падает без интернета до API.

## 5. Схема БД

```sql
users(id, email, password_hash, name, role ENUM('shipper','carrier','analyst'), company)

checkpoints(id, name, kind ENUM('sea_port','road_border','rail_border','road_hub'),
            lat, lng, capacity_per_hour INT)

cargo_requests(id, shipper_id→users, cargo_type ENUM('container','bulk','liquid','general'),
               weight_t NUMERIC, origin TEXT, destination TEXT, desired_date DATE,
               status ENUM('open','accepted','slot_booked','done','cancelled'),
               created_at)

assignments(id, request_id→cargo_requests UNIQUE, carrier_id→users,
            truck_plate TEXT, accepted_at)

slots(id, checkpoint_id→checkpoints, starts_at TIMESTAMP, ends_at TIMESTAMP,
      capacity INT, booked_count INT DEFAULT 0)
      -- генерируются сидом на 7 дней вперёд, по часу

bookings(id, slot_id→slots, assignment_id→assignments, qr_token UUID UNIQUE,
         status ENUM('booked','arrived','passed','no_show'), created_at)

traffic_history(id, checkpoint_id→checkpoints, ts TIMESTAMP, trucks_count INT,
                cargo_type, direction ENUM('export','import','transit'))
      -- 90 дней почасовой синтетики, основа аналитики и прогноза
```

Бизнес-правила:
- Бронь уменьшает свободную ёмкость слота; при booked_count == capacity слот недоступен (проверка в транзакции).
- QR-токен — UUID, публичная страница /pass/{qr_token} БЕЗ авторизации (чтобы судья отсканировал телефоном).
- Заявку может принять только carrier; слот бронируется только при принятой заявке.

## 6. Генератор синтетики (seed.py) — критично для правдоподобия

- 90 дней почасовой истории по каждому узлу.
- Паттерны: пик пн–чт днём, спад ночью и в вс; порт Актау — самый загруженный; недельная сезонность + шум ±20%; 3–4 «аномальных» дня (всплеск/обвал) — их потом найдёт AI-сводка, это запланированный wow-момент.
- Калибровка: суммарный годовой тоннаж порядка публичных 12 млн т через порт.
- 25–30 перевозчиков, 12–15 отправителей с казахстанскими названиями компаний, 60–80 заявок в разных статусах, 15–20 активных броней.
- Слоты на 7 дней вперёд для всех узлов.
- Сид должен быть идемпотентным (повторный запуск не дублирует).

## 7. API (минимальный список)

```
POST /api/auth/login
GET  /api/checkpoints                      ← + текущая загрузка (last hour из history)
GET  /api/checkpoints/{id}/forecast?days=7

POST /api/requests                         (shipper)
GET  /api/requests?status=open             (carrier feed) / ?mine=1 (shipper)
POST /api/requests/{id}/accept             (carrier)

GET  /api/slots?checkpoint_id=&date=
POST /api/bookings                         (carrier; body: slot_id, assignment_id)
GET  /pass/{qr_token}                      ← публичная HTML-страница пропуска

GET  /api/analytics/overview               ← карточки: всего т/день, активные заявки, средняя загрузка
GET  /api/analytics/timeseries?checkpoint_id=&days=90
GET  /api/analytics/breakdown?by=cargo_type|direction|checkpoint
POST /api/analytics/ai-summary
```

## 8. Фронтенд

Три зоны после логина по роли. Стек: React + Vite, react-leaflet (OpenStreetMap тайлы), Recharts, qrcode.react.

Дизайн — без шаблонного AI-вида. Направление: «диспетчерская транзита».
- Палитра: глубокий стальной фон #14181F, панели #1C222B, текст #E8ECF1, акцент — янтарный #F5A623 (цвет сигнальных огней/портовых кранов), статусы: загрузка низкая #3FB68B / средняя #F5A623 / высокая #E2574C.
- Шрифты: заголовки и цифры — моноширинный или полумоно (например, 'JetBrains Mono' / 'IBM Plex Mono') — даёт характер табло диспетчерской; основной текст — 'Inter'.
- Сигнатурный элемент: большая карта-табло у аналитика, узлы пульсируют цветом загрузки; вертикальная панель «лента событий» (новая заявка, бронь, прохождение) обновляется при действиях — на демо создаёшь заявку в одной вкладке, она появляется в ленте аналитика в другой. Это главный wow.
- Никаких лишних анимаций; аккуратные числа, единицы измерения (т, маш/час).
- Интерфейс на русском.

## 10. Презентация (8–10 слайдов, маппинг на требования кейса)

1. Проблема: 12+ млн т грузов/год, 3 000+ перевозчиков — координация через звонки и WhatsApp; очереди, простои, ноль аналитики у акимата.
2. Решение одним экраном: одна платформа, три роли, путь груза виден всем.
3. Живое демо (это 60% времени питча): заявка → принятие → слот → QR на экране (судья сканирует) → дашборд аналитика, где всё это отразилось → AI-сводка.
4. Данные: узлы и маршруты реальные (карта), трафик синтетический, откалиброван на публичную статистику; как подключаются реальные данные (API, датчики АСИ).
5. Экономика: простой фуры ≈ 10–15 тыс ₸/час (аренда+водитель+топливо). Слотирование срезает ожидание с ~4 ч до ~1 ч. 200 машин/день × 3 ч × 12 тыс ₸ ≈ 7,2 млн ₸/день экономии по региону (~2,6 млрд ₸/год). Цифры подписать как консервативную оценку.
6. Чем мы не дублируем госсистемы: АСТАНА-1/КТЖ — таможня и ж/д; мы — координация СМБ и авто, интегрируемся, а не конкурируем.
7. Что в MVP / что за рамками: за рамками — оплата, мобильные приложения, интеграция с погранслужбой, ML-прогноз на реальных данных.
8. Команда + следующий шаг (пилот с одним узлом / ассоциацией перевозчиков).
